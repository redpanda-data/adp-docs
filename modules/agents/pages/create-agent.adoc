== Prerequisites

* An ADP BYOC environment.
* At least one xref:ai-gateway:configure-provider.adoc[LLM provider configured] in AI Gateway.
* Optional: One or more xref:mcp:overview.adoc[MCP servers] registered in ADP if you want the agent to call tools.
* A system prompt prepared in advance (see xref:agents:system-prompts.adoc[System prompt best practices]).

== Open the create wizard

. Sign in to ADP and open the *AI Agents* page.
. Click *Create agent*. The create wizard opens.

The wizard validates each step before you advance, and the *Review* step lets you jump back to earlier steps before you submit.

== Configure agent basics

On the *Basics* step, configure identity fields that appear in the registry list, transcripts, and audit logs.

. Enter a *Display name*. Required. Up to 128 characters. Shown in the registry list and detail header (for example, `Customer Support Agent`).
. Review the auto-derived *Agent ID*. The UI generates a DNS-1123 / AIP-122 compliant slug from the display name (for example, `customer-support-agent`). The field stays in sync until you type into it manually; after that, the slug stops updating. The agent ID is immutable after creation.
. Optionally, add a *Description*. Up to 1024 characters.
. Optionally, add *Tags* as key/value pairs. Up to 50 pairs per agent. Tag values are limited to 256 characters.

== Choose a model and execution settings

On the *Model* step, select the glossterm:large language model (LLM)[] provider and model the agent uses to interpret user intent and decide which tools to invoke.

. Select an *LLM provider* from the providers configured on this cluster. Supported provider types:
+
** OpenAI (GPT models)
** Anthropic (Claude models)
** Google AI (Gemini models)
** AWS Bedrock (Claude, Llama, Titan, Nova through your AWS account)
** OpenAI-compatible (vLLM, Ollama, LM Studio, LocalAI, Together, Groq, OpenRouter)

. Select a *Model*. Pick from the models configured on the selected provider. If the provider has no allowlist, type the model identifier directly (for example, `gpt-4o-mini`).
+
For model selection guidance, see xref:agents:architecture-patterns.adoc#model-selection-guide[Model selection guide].

. Optionally, expand *Advanced* to set *Max iterations* (0-200, default 0). This is the upper bound on agent reasoning loops per request. Leave at 0 to use the runtime default.
+
Each iteration consumes tokens and adds latency. Increase this for exploratory queries with deep tool chaining; lower it for simple single-tool lookups. For the cost, capability, and latency trade-off, see xref:agents:concepts.adoc[].

For external model specifications and pricing:

* link:https://platform.openai.com/docs/models[OpenAI models^]
* link:https://docs.anthropic.com/claude/docs/models-overview[Anthropic Claude models^]
* link:https://ai.google.dev/gemini-api/docs/models[Google Gemini models^]
* link:https://docs.aws.amazon.com/bedrock/latest/userguide/models-supported.html[AWS Bedrock models^]

== Write the system prompt

On the *Behavior* step, write the system prompt that defines the agent's role, responsibilities, constraints, and output format.

. Enter a *System prompt*. Optional. Up to 16,384 characters.
. A typical prompt defines:
+
* Agent role and responsibilities
* Available tools (matches the tools you select in Step 4)
* Safety rules and constraints
* Expected output format

Example structure:

[,text]
----
You are an [agent role].

Responsibilities:
- [Task 1]
- [Task 2]

Available tools:
- [tool_name]: [description]

Never:
- [Constraint 1]
- [Constraint 2]

Response format:
- [Format guideline]
----

For full guidance on writing effective prompts, see xref:agents:system-prompts.adoc[System prompt best practices].

== Connect MCP servers

On the *Tools* step, connect the glossterm:Model Context Protocol (MCP)[] servers the agent can call. This step is optional. You can change the list at any time after creation.

. In the *MCP servers* multi-select, pick one or more servers registered in ADP. Each selected server exposes its full tool list to the agent.
. The agent calls tools based on the LLM's interpretation of the system prompt and the user request. Keep the system prompt aligned with the tools you select so the model knows what's available.

TIP: Apply the principle of least privilege. Connect only the servers the agent needs.

For registering MCP servers, see xref:mcp:create-server.adoc[Create an MCP server].

== Add subagents

The *Subagents* step is optional. Subagents are internal specialists within a single agent. Each subagent has its own name, description, system prompt, and MCP tools, but all subagents share the parent agent's model and execution settings.

. Click *Add subagent*.
. Configure the subagent:
+
* *Name*: Required. Up to 63 characters. Must start with a lowercase letter and contain only lowercase letters, digits, and hyphens (for example, `order-agent`). The name is also a DNS-1123 / AIP-122 compliant identifier.
* *Description*: Optional. Up to 1024 characters.
* *System prompt*: Up to 16,384 characters. Domain-specific instructions for what the subagent does.
* *MCP tools*: Select the tools this subagent can access.

The root agent orchestrates and delegates work to the appropriate subagent based on the request.

For multi-agent design patterns, see xref:agents:architecture-patterns.adoc[Agent architecture patterns].

== Review and create

On the *Review* step, verify all settings before submission.

. Review the summary. Each section has an *Edit* link that returns you to that step.
. Click *Create agent*. ADP provisions the agent and the service account that authenticates it.
. When the agent reaches *Running*, ADP shows its HTTP endpoint URL on the *Overview* tab. Copy the value from the *Endpoint* field.

You can use this URL to call the agent programmatically or integrate it with external systems. See xref:agents:integration-overview.adoc[].

== Configure A2A discovery metadata (optional)

A2A discovery metadata lets external systems find and invoke your agent through capability-based discovery. Configure this after creation, on the agent's *Overview* tab.

For agent card design guidance, see link:https://agent2agent.info/docs/guides/create-agent-card/[Create an Agent Card^].

. Open the agent's *Overview* tab.
. Configure identity fields:
+
* *Icon URL*: A publicly accessible image URL (recommended: 256x256 PNG or SVG).
* *Documentation URL*: Link to comprehensive agent documentation.

. Configure provider information:
+
* *Organization*: Your organization or team name.
* *URL*: Website or contact URL.

. Add *Skills* to describe what the agent can do for capability-based discovery. For each skill:
+
* *Skill ID* (required): Lowercase letters, digits, and hyphens (for example, `fraud-analysis`).
* *Skill name* (required): Human-readable name displayed in agent directories.
* *Description* (required): What the skill does and when to use it. Be specific about inputs, outputs, and use cases.
* *Tags* (optional): Categorization terms (for example, `fraud`, `security`, `finance`).
* *Examples* (optional): Sample queries demonstrating how to invoke this skill.

. Save your changes. The updated metadata appears immediately at `\https://<your-agent-url>/.well-known/agent-card.json`.

For more about agent cards and how they're consumed, see xref:agents:a2a-concepts.adoc#agent-cards[Agent cards].

== Test your agent

Test the agent through the *Transcripts* tab on the agent's detail page. Transcripts record every conversation against the agent (prompt, tool calls, tool outputs, final response) so you can verify the agent selects the right tools, follows the system prompt, and returns the expected output format.

. Open the agent's *Transcripts* tab.
. Send a test prompt through the endpoint (see xref:agents:integration-overview.adoc[] for SDK options).
. Inspect the transcript to verify the agent:
+
* Selects appropriate tools
* Follows the system prompt's constraints
* Returns the expected output format

. Iterate on the system prompt or tool selection as needed.

For detailed observability strategies, see xref:agents:monitor.adoc[].

== Example configurations

=== Simple query agent

* *Model*: GPT-5 Mini (fast, cost-effective)
* *Tools*: Single MCP server exposing a `get_orders` tool
* *Max iterations*: 10
* *Use case*: Customer order lookups

=== Complex analytics agent

* *Model*: Claude Sonnet 4.5 (balanced)
* *Tools*: Multiple servers for data query, aggregation, and formatting
* *Max iterations*: 30
* *Use case*: Multi-step data analysis

=== Multi-agent orchestrator

* *Model*: Claude Opus 4.5 (advanced reasoning)
* *Tools*: Tools the root agent delegates to subagents
* *Subagents*: `order-agent`, `inventory-agent`, `customer-agent`
* *Max iterations*: 20
* *Use case*: E-commerce operations
